> 개인 프로젝트 제출용 v0.6 (2026-07-12)

## 1. 개요

| 항목 | 내용 |
|---|---|
| 서비스명 | ⚠️ 미정 (가칭: 대장관리 플랫폼 / 후보: 팀 업무 데이터 관리 플랫폼, 업무 대장 통합 관리 플랫폼 — 통계·지도·AI 리포트로의 확장 로드맵을 고려하면 '대장'보다 '업무 데이터' 중심 명칭도 검토 가치 있음) |
| 한 줄 소개 | 흩어진 엑셀 대장을 하나의 데이터베이스로 통합하여, 팀의 업무 데이터에 대한 **단일 원본(Single Source of Truth)**을 구축하는 내부망 전용 웹 플랫폼 |
| 대상 사용자 | 1단계: 팀원 5명 (파일럿) → 이후 과 전체 확장 |
| 배포 형태 | 내부망 전용 (담당자 PC를 서버로 사용, 외부 인터넷 미노출) |

### 서비스가 제공하는 가치

이 시스템의 본질은 "엑셀을 웹으로 옮기는 것"이 아니라, **팀 업무 데이터의 관리 체계를 바꾸는 것**이다.

- **단일 원본(Single Source of Truth)**: 대장의 원본은 DB 한 곳에만 존재한다. 사본이 만들어질 이유가 사라지므로 "어느 파일이 최신인가"라는 질문 자체가 없어진다
- **항상 최신 데이터 공유**: 관리자가 수정하는 순간, 모든 팀원은 브라우저 새로고침만으로 최신 상태를 조회한다. 파일 전달·회람 절차가 불필요하다
- **버전 충돌 원천 제거**: 수정 권한을 가진 계정만 데이터를 변경할 수 있으므로, 여러 사본이 서로 다르게 수정되는 상황이 구조적으로 발생하지 않는다
- **책임 소재의 명확화**: 모든 변경에 수정자와 시각이 기록되어, 공공기관 업무에 요구되는 이력 관리가 자동으로 이루어진다

## 2. 배경 및 문제 정의

### 현재 방식 (엑셀 파일 기반)의 구조적 한계

업무 대장을 엑셀 파일로 관리하는 현재 방식에서는 다음 문제가 반복된다.

**사례 1 — "어느 파일이 최신인가?"**
공유폴더에 `대장_최종.xlsx`, `대장_최종_수정.xlsx`, `대장_0710.xlsx`처럼 유사한 이름의 파일이 공존한다. 최신본을 확인하기 위해 담당자에게 메신저나 구두로 문의해야 하고, 담당자가 부재중이면 업무가 지연된다.

**사례 2 — 잘못된 버전으로 업무 처리**
팀원이 구버전 파일을 최신본으로 오인하여 민원 응대나 보고 자료에 반영하는 경우, 사후에 정정 절차가 필요하고 대외 신뢰도에도 영향을 미친다.

**사례 3 — 동시 수정과 덮어쓰기**
두 사람이 각자 파일을 열어 수정한 뒤 저장하면 한쪽의 수정 내용이 유실된다. 누가 언제 무엇을 고쳤는지 추적할 수단이 없어, 문제가 발견되어도 원인 규명이 어렵다.

**사례 4 — 인수인계의 어려움**
담당자 변경 시 "어떤 파일이 원본이고, 어떤 규칙으로 관리해왔는지"가 문서화되어 있지 않아, 인수인계가 개인의 기억과 구두 설명에 의존한다.

### 해결 방향

**원본 데이터를 한 곳(DB)에 두고**, 열람은 웹 브라우저로 팀원 누구나, 수정은 권한을 부여받은 계정만 가능하도록 역할을 분리한다. 모든 변경은 자동으로 이력에 남긴다.

### Before / After — 업무 흐름 비교

**Before (엑셀 파일 기반)**
```
엑셀 파일 → 공유폴더 → 각자 복사 → 개별 수정 → 버전 충돌
                                        ↓
                              "최신본이 뭐예요?" 메신저 문의 → 업무 지연
```

**After (플랫폼 도입)**
```
브라우저 접속 → 로그인 → 항상 최신 데이터 조회
                          ↓ (write 권한자만)
                     수정 → DB 즉시 반영 → 변경 이력 자동 기록 → 자동 백업
```

버전 확인·파일 전달·수동 취합이라는 중간 절차 자체가 사라진다.

⚠️ 실제 대장 종류(무슨 대장인지)를 확정하고, 위 사례 중 실제 경험한 사례로 교체·보강하면 설득력이 더 올라감

## 3. 프로젝트 목표

1. 지금까지 학습한 내용(React, FastAPI, DB, AI 협업 개발)을 활용해 **작동 가능한 웹 서비스 MVP** 구현
2. 실제 업무에 즉시 투입 가능한 도구 제작 — 과제용 데모에 그치지 않는 실용성
3. 생성형 AI(Claude + Cursor)와 협업하는 개발 프로세스(SDD: 명세 우선 개발) 실천

## 4. MVP 범위 (1단계)

### 포함 (In Scope)
- 개인 계정 로그인 (ID/PW) + 역할 기반 권한 (read / write / admin)
- 계정 관리 — 권한 관리자가 계정 생성·역할 지정·비밀번호 초기화·비활성화
- 대장 목록 조회 (검색·정렬 포함)
- 대장 항목 추가 / 수정 / 삭제 (`write` 이상 계정 전용, 삭제는 Soft Delete)
- 변경 이력 자동 기록 (audit_log)
- 엑셀 → DB 초기 데이터 이관 스크립트
- SQLite 자동 백업

### 제외 (Out of Scope — 이후 단계)
- 비밀번호 자가 재설정(이메일 인증 등) — 1단계는 관리자가 초기화
- 검색 고도화(다중 조건 필터, 엑셀 다운로드 등)
- 시설관리 업무 지도
- 탐색적 통계분석 / 대시보드
- 동시 다중 편집

## 5. 사용자 및 권한 모델

**개인 계정 + 역할(role) 기반 접근 제어.** 사내에서 이미 운영 중인 팀 단위 프로그램과 동일한 패턴 — 권한 관리자가 계정을 발급하고 계정별로 역할을 지정, 사용자는 발급받은 ID/PW로 브라우저에서 로그인.

| 역할 | 인원 (1단계) | 할 수 있는 것 |
|---|---|---|
| 읽기 전용 (`read`) | 팀원 | 대장 조회, 검색 |
| 읽기+쓰기 (`write`) | 자료관리자 (필요 시 복수 지정 가능) | 조회 + 추가/수정/삭제 |
| 권한 관리자 (`admin`) | 1명 (본인, 후임 인계 가능) | 위 전체 + 계정 생성/역할 지정/비밀번호 초기화/비활성화 |

- **계정 발급 방식**: 회원가입 화면 없음. 권한 관리자가 계정을 생성해 ID/초기 PW를 전달 (초대제)
- **권한 강제 지점**: 프론트엔드가 아닌 **백엔드(FastAPI) 엔드포인트에서 role 검증** — 화면을 우회해도 수정 불가
- **설계 의도**: 자료 관리 권한이 실제 업무에서 분산되어 있으므로, 처음부터 계정 단위로 role을 부여해 "누가 수정 권한이 있는지"를 선택적으로 관리. 과 전체 확장 시에도 계정 추가만으로 대응 가능

### 사용자 흐름 (User Flow)

**열람자 (read)**
```
로그인 → 대장 목록 → 검색/정렬 → 상세 조회
```

**자료관리자 (write)**
```
로그인 → 대장 목록 → 항목 추가/수정/삭제 → 저장
      → DB 즉시 반영 + 변경 이력 자동 기록 → 팀원은 다음 조회 시 최신 데이터 확인
```

**권한 관리자 (admin)**
```
로그인 → 계정 관리 화면 → 계정 생성 / 역할 변경 / 비밀번호 초기화 / 비활성화
      → 처리 내역이 관리 이력에 기록
```

## 6. 기술 스택 및 아키텍처

```
[팀원 브라우저] ──내부망──> [담당자 PC]
                              ├─ FastAPI (백엔드, 포트 ⚠️미정)
                              ├─ React + Vite + TS (프론트엔드 빌드 결과물)
                              └─ ledger.db (SQLite 파일)
```

| 계층 | 선택 | 선택 이유 |
|---|---|---|
| 프론트엔드 | React + Vite + TypeScript | 컴포넌트 기반이라 화면 재사용·확장이 용이하고, SPA 구조라 조회·검색 중심 업무에서 페이지 이동 없이 빠른 사용 경험 제공. TypeScript로 대장 필드 타입을 강제해 데이터 입력 오류를 컴파일 단계에서 차단. 과제5에서 구현한 CRUD 화면을 재사용해 개발 기간 단축 |
| 백엔드 | FastAPI (Python) | MyDataInsight에서 사용 경험, 단일 프로세스로 운영 단순 |
| DB | SQLite | 서버 설치 불필요, 파일 1개 = 백업 간편, 쓰기 주체가 소수라 동시성 문제 없음. **향후 동시 접속자 증가 또는 과 전체 확대 시 PostgreSQL로 이전할 수 있도록 DB 접근 계층을 별도 모듈로 분리하여 설계** |
| 초기 데이터 | pandas (엑셀 → SQLite 이관) | 기존 데이터 전처리 경험 활용 |
| 개발 도구 | Cursor(구현) + Claude(설계·리뷰) | Generator-Critic 워크플로 |

### 데이터 흐름

**운영 시 (조회/수정)**
```
React (사용자 조작)
  ↓ HTTP 요청 (내부망)
FastAPI (인증·권한 검증 → 비즈니스 로직)
  ↓ SQL
SQLite (ledger.db 읽기/쓰기 + 변경 이력 기록)
  ↓ 결과 반환
FastAPI → React (화면 갱신)
```

**초기 데이터 이관 (1회성)**
```
기존 엑셀 파일
  ↓ pandas (읽기·정제·컬럼 매핑)
SQLite (ledger 테이블 적재)
  ↓ 검증 (건수·필드 대조)
운영 시작
```

### Supabase 등 BaaS를 쓰지 않는 이유 (기획 판단 근거)
- 내부망 폐쇄 운영 요건 → 외부 클라우드 사용 불가
- 셀프호스팅 Supabase는 Docker 기반 다중 컨테이너 구성으로, 업무 PC 설치 승인·운영 부담이 큼
- 단일 프로세스 + 단일 파일 DB가 유지보수(백업=파일복사, 배포=폴더복사) 측면에서 우위
- 단, DB 접근 코드를 한 모듈에 격리해 향후 PostgreSQL 등으로 교체 가능하게 설계

## 7. API 설계 초안 (REST)

| 메서드 | 경로 | 기능 | 필요 권한 |
|---|---|---|---|
| POST | `/login` | 로그인, 인증 토큰/세션 발급 | 없음 |
| POST | `/logout` | 로그아웃 | 로그인 |
| GET | `/ledger` | 대장 목록 조회 (검색·정렬 파라미터) | read 이상 |
| GET | `/ledger/{id}` | 대장 항목 상세 조회 | read 이상 |
| POST | `/ledger` | 대장 항목 추가 | write 이상 |
| PATCH | `/ledger/{id}` | 대장 항목 수정 (변경 필드만 전송하는 부분 수정 — 대장 수정은 대부분 일부 필드 갱신이므로 전체 교체(PUT)보다 PATCH가 적합) | write 이상 |
| DELETE | `/ledger/{id}` | 대장 항목 삭제 (**Soft Delete**) | write 이상 |
| GET | `/users` | 계정 목록 조회 | admin |
| POST | `/users` | 계정 생성 | admin |
| PATCH | `/users/{id}` | 역할 변경·비밀번호 초기화·비활성화 | admin |
| GET | `/audit-log` | 변경 이력 조회 | admin |

### Soft Delete 설계

DELETE 요청은 실제로 행을 지우지 않고 `is_deleted` 플래그만 변경한다.

- **이유 1 (복구 가능성)**: 실수로 삭제해도 관리자가 복원 가능
- **이유 2 (이력 보존)**: 공공기관 업무 특성상 "삭제된 기록이 존재했다는 사실" 자체가 이력으로 남아야 함
- 목록 조회 시 기본적으로 `is_deleted = 0`인 행만 노출

## 8. 데이터 모델 (초안)

### 8-1. users (계정)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | TEXT (UUID) | 기본키 |
| username | TEXT | 로그인 ID (고유) |
| password_hash | TEXT | bcrypt 해시 (평문 저장 금지) |
| role | TEXT | `read` / `write` / `admin` |
| is_active | INTEGER | 계정 비활성화 플래그 (퇴직·이동 시 삭제 대신 비활성화) |
| failed_login_count | INTEGER | 로그인 실패 누적 (잠금 정책용) |
| last_login_at | TEXT | 최종 로그인 시각 — 관리자의 휴면 계정 식별용 |
| created_at | TEXT | 생성 시각 |

### 8-2. ledger (대장)

기준: 기존 "공유주차장 이용현황 분석기" 앱의 이용자 명부 구조(11개 필드)를 그대로 이관. 단, 등록기간은 원본에서 "시작일~종료일" 문자열로 결합 저장되던 것을 조회·정렬이 가능하도록 시작일/종료일 컬럼으로 분리.

| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | TEXT (UUID) | 기본키. 원본의 '연번'(자동 부여 순번)은 화면 표시용으로만 유지하고 PK 역할은 id가 대체 |
| name | TEXT | 이름 |
| gender | TEXT | 성별 (`남`/`여`/`기타`) |
| address | TEXT | 주소 |
| phone | TEXT | 연락처 |
| car_number | TEXT | 차량번호 — 원본에서 유일한 필수 항목, DB에서도 NOT NULL |
| car_model | TEXT | 차종 |
| category | TEXT | 구분 (`거주자`/`회사원`/`기타`) |
| registered_at | TEXT (날짜) | 등록일 |
| period_start | TEXT (날짜) | 등록기간 시작일 — 원본의 결합 문자열을 분리 |
| period_end | TEXT (날짜) | 등록기간 종료일 — 만료 예정 조회·정렬을 위해 날짜 타입으로 분리 |
| note | TEXT | 비고 |
| is_deleted | INTEGER | Soft Delete 플래그 |
| updated_by | TEXT | 최종 수정자 username |
| created_at | TEXT | 생성 시각 |
| updated_at | TEXT | 최종 수정 시각 |

⚠️ 위반이력(위반자 관리) 기능은 원본 앱에 존재하지만 이번 MVP 범위(대장만)에서는 제외 — 이후 별도 대장/기능으로 확장 시 재검토

### 8-3. audit_log (변경 이력)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | INTEGER | 기본키 (자동 증가) |
| table_name | TEXT | 변경된 테이블 (`ledger` / `users`) |
| record_id | TEXT | 변경된 행의 id |
| action | TEXT | `create` / `update` / `delete` / `restore` / `account_change` |
| result | TEXT | `SUCCESS` / `FAIL` — 권한 부족으로 거부된 시도 등도 기록 |
| old_value | TEXT (JSON) | 변경 전 값 (FAIL 행은 비움) |
| new_value | TEXT (JSON) | 변경 후 값 (FAIL 행은 시도된 요청 내용만 기록) |
| username | TEXT | 수행자 |
| ip_address | TEXT | 수행 PC의 내부망 IP — 계정 공유·비인가 사용 의심 시 어느 PC에서 수행됐는지 판별 |
| created_at | TEXT | 수행 시각 |

(user_agent는 미채택 — 전 사용자가 관용 PC + 표준 브라우저 환경이라 판별 정보로서의 가치가 낮음)

**audit_log가 필요한 이유**: 공공기관 업무 데이터는 "현재 값"만큼 "누가, 언제, 무엇을, 어떻게 바꿨는가"가 중요하다. 감사·민원 대응·업무 인수인계 시 변경 경위를 소명할 수 있어야 하며, 이를 사람의 기억이 아닌 시스템이 자동 기록하도록 한다. 계정 관리 작업(권한 변경, 비밀번호 초기화 등 admin 행위)도 동일하게 기록하여 관리자 책임 소재를 명확히 한다.

**SUCCESS / FAIL 기록 규칙**: 성공 행은 변경 전후 값을 온전히 기록하는 "데이터 변경 이력"이고, FAIL 행은 권한 부족 등으로 거부된 시도를 남기는 "보안 이벤트 기록"이다 (예: read 계정이 수정 API를 호출한 경우). 두 성격이 다르므로 FAIL 행은 old_value를 비우고 시도 내용만 남긴다. 반복적인 FAIL 기록은 계정 오지정이나 비인가 사용 시도를 조기에 발견하는 단서가 된다.

### 8-4. 테이블 관계도 (ERD)

```
users (계정)
  │ 1
  │
  │ N (username 참조)
  ├────────────► ledger.updated_by      ── 대장 각 행의 최종 수정자
  │
  │ N (username 참조)
  └────────────► audit_log.username     ── 변경 이력의 수행자

ledger (대장)
  │ 1
  │ N (record_id 참조, table_name='ledger')
  └────────────► audit_log              ── 해당 행의 모든 변경 이력

audit_log는 users의 변경(계정 생성·권한 변경 등)도 기록
  (record_id 참조, table_name='users')
```

- 관계 요약: **users 1 : N ledger(수정자)**, **users 1 : N audit_log(수행자)**, **ledger 1 : N audit_log(변경 이력)**
- audit_log는 다형적으로 두 테이블(`ledger`, `users`)의 변경을 모두 수용 — `table_name` + `record_id` 조합으로 대상 식별

## 9. 화면 구성 (초안)

1. **로그인 화면** — ID/PW 입력
2. **대장 목록 화면** — 표 형태, 검색창, 정렬. 로그인 후 기본 진입 화면
3. **관리자 화면** (`admin` 전용) — 계정 목록, 계정 생성, 역할 변경, 비밀번호 초기화, 계정 비활성화, 변경 이력(audit_log) 조회
4. (선택) 상태 표시 — 마지막 갱신 시각, 총 건수

### 권한별 화면 차이

같은 대장 목록 화면이라도 로그인한 계정의 role에 따라 노출되는 UI가 달라진다.

| role | 대장 목록 화면 | 계정 관리 메뉴 |
|---|---|---|
| `read` | 조회·검색·정렬만 가능. 추가/수정/삭제 버튼 자체가 렌더링되지 않음 | 미노출 |
| `write` | 조회 + 행 추가·수정·삭제 버튼 표시 | 미노출 |
| `admin` | `write`와 동일 | 상단 메뉴에 계정 관리·변경 이력 노출 |

단, 화면에서 버튼을 숨기는 것은 사용 편의를 위한 것이며, **실제 권한 차단은 백엔드 API에서 수행**한다 (10번 보안 설계 참조).

## 10. 보안 설계

| 항목 | 내용 |
|---|---|
| 비밀번호 저장 | bcrypt 해시 (평문 저장 금지) |
| 인증 방식 | **Session 기반으로 확정.** 근거: ① 서버 1대 구조라 JWT의 장점(무상태·다중 서버 확장)을 쓸 일이 없음 ② 세션은 서버에서 즉시 무효화 가능 — 계정 비활성화 시 해당 사용자의 접속을 그 자리에서 차단 (JWT는 만료 전까지 유효) ③ 내부망 + 5명 규모에서 구현·유지보수가 가장 단순 |
| 인증 만료 | 세션/토큰에 만료 시간 설정, 만료 시 재로그인 요구 |
| 로그인 실패 제한 | 연속 실패 N회 시 계정 잠금 (관리자가 해제) — 무차별 대입 방지 |
| 권한 검증 위치 | 모든 쓰기/관리 API에서 백엔드가 role 검증 (프론트 우회 불가) |
| 관리자 작업 기록 | 계정 생성·권한 변경·비밀번호 초기화 등 admin 행위를 audit_log에 기록 |
| 전송 구간 암호화 | 1단계(내부망 폐쇄 환경)는 HTTP로 운영, **HTTPS는 과 전체 확장 또는 외부 접점 발생 시 적용 예정** |
| 개인정보 | 과제 제출·발표용 데모는 **더미 데이터만 사용**. 실 데이터 투입은 조직 정보보안 정책 검토 후 진행 |

### 시퀀스 다이어그램 — 대장 수정 (권한 검증 + 이력 기록)

시스템의 핵심 동작인 "권한 검증을 거친 수정과 자동 이력 기록"의 처리 순서.

```
자료관리자          React            FastAPI                SQLite
   │                 │                 │                     │
   │ 수정 버튼 클릭   │                 │                     │
   ├────────────────►│                 │                     │
   │                 │ PATCH /ledger/{id}                    │
   │                 │ (인증 토큰/세션 포함)                   │
   │                 ├────────────────►│                     │
   │                 │                 │ ① 인증 확인          │
   │                 │                 │    (만료 시 401 반환) │
   │                 │                 │ ② role 검증          │
   │                 │                 │    (read면 403 반환)  │
   │                 │                 │ ③ 변경 전 값 조회     │
   │                 │                 ├────────────────────►│
   │                 │                 │◄────────────────────┤
   │                 │                 │ ④ 트랜잭션 시작       │
   │                 │                 │    ledger 갱신        │
   │                 │                 │    + audit_log 기록   │
   │                 │                 │    (old/new/수행자/IP)│
   │                 │                 ├────────────────────►│
   │                 │                 │ ⑤ 커밋               │
   │                 │◄────────────────┤                     │
   │                 │ 200 OK + 갱신 데이터                   │
   │ 화면 갱신        │                 │                     │
   │◄────────────────┤                 │                     │
```

- **④가 트랜잭션인 이유**: 대장 갱신과 이력 기록은 반드시 함께 성공하거나 함께 실패해야 함. 갱신만 되고 이력이 안 남는 상황을 DB 수준에서 차단
- 조회(GET)는 ①②만 거치고 곧바로 데이터 반환

## 11. 백업 및 운영 정책

SQLite는 DB 전체가 파일 1개이므로 백업 절차가 단순하다는 것이 강점이며, 이를 자동화한다.

| 구분 | 정책 |
|---|---|
| 일일 백업 | 매 영업일 지정 시각에 `ledger.db`를 백업 폴더로 자동 복사 (Windows 작업 스케줄러 + 스크립트) |
| 보관 주기 | 일일 백업은 **최근 7일분 보관** (7일 경과분 자동 삭제) |
| 월간 백업 | 매월 1회분은 별도 폴더에 장기 보관 |
| 백업 위치 | 서버 PC 내 별도 폴더 + ⚠️ 가능하면 부서 공유폴더 등 2차 위치 병행 (서버 PC 고장 대비) |
| 복구 절차 | 백업 파일을 원위치에 복사하는 것으로 완료 — 복잡한 복구 도구 불필요 |
| 가용성 | 담당자 PC 상시 가동 필요 — 파일럿 단계에서는 업무시간 내 가동으로 충분 |

기존에 법령레이더 등에서 Windows 작업 스케줄러 기반 자동화를 운영한 경험이 있어, 동일한 방식(스케줄러 + 스크립트)으로 추가 도구 설치 없이 구현 가능하다.

## 12. 단계별 로드맵

| 단계 | 범위 | 비고 |
|---|---|---|
| 1단계 (MVP) | 팀원 5명 파일럿 — 로그인·권한, 대장 CRUD, 변경 이력, 백업 | 이번 개인 프로젝트 |
| 2단계 | 검색 고도화 — 다중 조건 필터, 정렬 저장, 엑셀 다운로드 | 파일럿 피드백 반영 |
| 3단계 | 권한 관리 고도화 — 과 전체 확장, 계정 일괄 등록, 접속 이력 | 사용자 확대 |
| 4단계 | 통계 기능 — 대장 데이터 기반 집계·탐색적 통계 화면 | |
| 5단계 | 지도 기능 — 시설관리 업무 지도 연동 | |
| 6단계 | AI 리포트 — 4~5단계에서 축적된 통계·지도 데이터를 로컬 LLM이 요약해 정기 보고 초안 자동 생성 | 기존 로컬 LLM 운용 경험 활용. 데이터가 내부망을 벗어나지 않는 로컬 모델 사용 |

각 단계는 이전 단계의 산출물을 기반으로 한다 — 대장(1~3단계)이 데이터를 쌓고, 통계·지도(4~5단계)가 그 데이터를 구조화하며, AI 리포트(6단계)는 구조화된 결과를 보고 형태로 변환한다.

## 14. 프로젝트 성공 기준 (Success Criteria)

MVP 완료 시점에 아래 항목을 실제로 시연·확인할 수 있으면 프로젝트 성공으로 판정한다. 각 항목은 파일럿 환경(팀원 5명, 내부망)에서 검증한다.

| 항목 | 성공 기준 |
|---|---|
| 접속 | 팀원 PC 브라우저에서 담당자 PC의 서비스에 접속 가능 |
| 로그인 | 발급된 계정으로 정상 로그인, 잘못된 PW 반복 시 계정 잠금 동작 |
| 조회 | 5명 동시 접속 상태에서 대장 목록 조회·검색 정상 동작 |
| CRUD | write 계정으로 추가/수정/삭제(Soft Delete) 정상 수행 |
| 권한 차단 | read 계정으로 수정 API 호출 시 거부(403) + FAIL 이력 기록 |
| 변경 이력 | 모든 수정에 대해 audit_log에 수행자·전후 값·IP가 자동 기록 |
| 데이터 이관 | 기존 엑셀 데이터가 건수·필드 누락 없이 DB로 이관 |
| 백업 | 지정 시각에 백업 파일 자동 생성, 백업 파일로 복구 성공 |

## 15. 기대 효과

정량 수치는 파일럿 운영 후 측정하며, 기획 단계에서는 정성적 효과를 제시한다.

- **최신 데이터 단일화**: "어느 파일이 최신인지" 확인하는 절차가 사라지고, 조회 시점의 데이터가 곧 원본이 됨
- **업무 처리 시간 감소**: 파일 탐색·버전 확인·담당자 문의에 소요되던 시간이 제거됨
- **버전 충돌 및 데이터 유실 제거**: 사본 기반 관리에서 발생하던 덮어쓰기 사고가 구조적으로 불가능해짐
- **자료관리 책임 명확화**: 수정 권한이 계정 단위로 부여되고 모든 변경이 기록되어, 데이터 품질에 대한 책임 소재가 분명해짐
- **감사 대응력 향상**: 변경 이력(audit_log)이 자동 축적되어, 경위 소명 요구에 즉시 대응 가능
- **인수인계 부담 경감**: 원본 위치·관리 규칙이 시스템에 내재되어, 담당자 교체 시 구두 설명 의존도가 낮아짐
- **확장 기반 확보**: 데이터가 DB로 구조화되므로, 이후 통계·지도·AI 분석 기능을 같은 기반 위에 얹을 수 있음
